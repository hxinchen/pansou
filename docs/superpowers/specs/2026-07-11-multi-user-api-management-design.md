# PanSou 多用户与 API 管理系统设计

## 1. 目标

将现有公开搜索页面改造成需要登录的用户门户，并保留 `/admin/` 作为管理员后台。管理员能够手工创建和分发账号；普通用户能够通过网页或个人 API Key 搜索，并只查看自己的调用日志和监控数据。

系统继续保持单个 Go 服务和 PostgreSQL，不引入 Redis、独立认证服务、API 网关、消息队列或独立日志服务。

## 2. 已确认的产品规则

- V1 为封闭式发号，不开放自行注册。
- 普通用户只使用资源搜索、API 管理、个人调用监控和账号设置。
- 管理员可以访问用户门户和 `/admin/`；普通用户访问管理接口返回 `403`。
- 每个用户最多拥有一个 API Key，用户或管理员可以重置，旧 Key 立即失效。
- 网页搜索和 API Key 搜索共用同一用户限额，默认 `3 RPS / 60 RPM`。
- 普通用户不能使用 `refresh=true`；强制实时搜索仅管理员和内部采集任务可用。
- 账号默认永久有效，管理员可以设置到期时间并随时启用或停用。
- API 调用明细保留 30 天。
- 普通用户只能查看自己的数据；管理员可以查看全部用户的完整调用明细，包括关键词和来源 IP。
- 管理员创建账号时，系统生成临时密码，用户首次登录必须修改。
- 前台和后台均实现桌面与移动端响应式布局。

## 3. 总体架构

### 3.1 访问入口

- `/pansou/`：用户门户。未登录时显示登录页，登录后默认进入资源搜索。
- `/admin/`：管理员后台。保留资源库、关键词、采集任务和数据概览，新增用户管理与 API 监控。
- `/api/search`：继续兼容现有 GET、POST 参数和响应结构，同时支持网页 JWT 和 `X-API-Key` 两种身份来源。
- `/api/auth/*`：网页登录、令牌校验、退出、首次改密和用户修改密码。
- `/api/user/*`：当前用户资料、API Key 管理、个人调用统计与日志。
- `/api/admin/*`：管理员专用的用户管理、全局调用统计和日志查询。

未配置 `DATABASE_URL` 时，服务继续保留现有单一部署的兼容行为，但不启用数据库用户、API Key、个人监控和用户管理接口。启用多用户门户的部署必须配置 PostgreSQL。

### 3.2 模块边界

- `UserStore`：用户、角色、状态、有效期、密码版本和限流配置的持久化。
- `AuthService`：密码登录、JWT 签发与验证、账号状态校验和首次改密。
- `APIKeyService`：Key 生成、哈希校验、重置和吊销。
- `RateLimitService`：按用户执行共享的秒级与分钟级限流。
- `UsageRecorder`：生成调用事件，并批量异步写入 PostgreSQL。
- `UsageQueryService`：为用户和管理员提供调用明细、趋势和聚合统计。
- 现有 `SearchProvider`、混合搜索、资源库和采集逻辑保持独立，不承担身份与计量职责。

请求先经过身份解析、账号状态校验、权限校验和限流，再进入现有搜索服务；搜索完成后记录状态、耗时和结果摘要。

## 4. 数据模型

### 4.1 `users`

主要字段：

- 数字主键 `id`
- `username` 与唯一的 `normalized_username`
- `password_hash`
- `role`，V1 取值为 `admin` 或 `user`
- `enabled`
- 可空的 `expires_at`
- `must_change_password`
- `auth_version`，用于使旧 JWT 失效
- `rps_limit`，默认 3
- `rpm_limit`，默认 60
- `rate_limit_disabled`，用于管理员配置不限流账号
- `last_login_at`
- `created_at`、`updated_at`、可空的 `deleted_at`

密码使用 bcrypt 强哈希保存。用户名归一化后唯一，软删除后默认不立即允许复用。

### 4.2 `api_keys`

主要字段：

- `id`
- 唯一的 `user_id`
- `key_prefix`，用于界面识别
- `key_hash`
- `created_at`、`last_used_at`、可空的 `revoked_at`

Key 使用密码学安全随机数生成，完整值只在创建或重置响应中出现一次。数据库只保存 SHA-256 哈希和非敏感前缀。

### 4.3 `api_request_logs`

主要字段：

- 大整数主键 `id`
- `request_id`
- `user_id`
- `auth_type`，取值为 `web` 或 `api_key`
- `method`、`endpoint`
- 搜索 `keyword`
- `status_code`
- `duration_ms`
- `result_count`
- `cache_status`
- `error_code`
- `source_ip`
- `user_agent`
- `created_at`

为用户与时间、状态码与时间、创建时间建立索引。清理任务每天分批删除 30 天前的明细，避免大事务阻塞在线查询。

## 5. 账号与凭证生命周期

1. 管理员创建用户并填写用户名、可选有效期和可选限额覆盖值。
2. 系统生成临时密码和首个 API Key，在成功对话框中只显示一次。
3. 用户首次登录只能进入修改密码流程，完成前不能搜索、使用用户面板或通过 API Key 调用搜索。
4. 修改或重置密码时递增 `auth_version`，所有旧 JWT 随即失效。
5. 用户可以在个人 API 页面重置自己的 Key；管理员也可以重置或吊销。
6. 停用、到期或软删除用户后，JWT 和 API Key 均不可继续使用。
7. 管理员不能停用或删除当前唯一的有效管理员，避免系统失去管理入口。

首次启用数据库账号体系且 `users` 为空时，通过部署环境变量创建唯一的初始管理员。现有 `AUTH_USERS` 可作为一次性兼容引导来源；用户表非空后，数据库账号成为唯一事实来源，环境变量不再覆盖密码或角色。

## 6. 鉴权与权限

JWT 包含用户 ID、用户名、角色、`auth_version`、签发时间和过期时间。每次受保护请求读取当前用户状态，验证角色、启用状态、到期时间和密码版本。

API Key 通过 `X-API-Key` 传递，只允许访问普通搜索接口。API Key 不能访问认证、用户管理、管理员、健康检查或强制刷新能力。

权限规则：

- 未认证访问用户或管理接口返回 `401`。
- 普通用户访问管理员接口返回 `403`。
- 停用、到期或版本失效的凭证返回 `401`，并使用稳定错误码区分原因。
- 普通用户提交 `refresh=true` 返回 `403`，不静默降级。
- 页面路由守卫只负责用户体验，后端中间件始终执行最终权限校验。

## 7. 限流

网页 JWT 与 API Key 均先解析为用户 ID，再进入同一个双层限流器：

- 秒级窗口默认最多 3 个请求。
- 分钟级窗口默认最多 60 个请求。
- 任一窗口超限即返回 `429 Too Many Requests`。
- 返回 `RateLimit-Limit`、`RateLimit-Remaining`、`RateLimit-Reset` 和 `Retry-After` 响应头；剩余量和重试时间以当前更严格的窗口为准。
- 管理员修改限额后清除该用户的旧限流状态，使新配置立即生效。
- 被拒绝的超限请求也进入调用日志，但不会执行搜索。

V1 限流状态保存在单个 Go 进程内，符合当前单实例部署。服务重启会清空当前窗口计数，这是可接受的 V1 行为。未来改为多实例时再替换为共享限流存储。

## 8. 调用日志与统计

所有已认证的网页搜索和 API Key 搜索均记录日志，用户面板可以按调用入口筛选。记录范围包括成功、搜索失败、参数错误、禁止强刷和限流请求。

日志通过有界内存缓冲区批量写入 PostgreSQL：

- 日志写入短暂失败不会改变已完成搜索的响应。
- 缓冲区满时优先保证业务请求，并记录服务端丢弃计数和错误日志。
- 服务正常退出时尝试在限定时间内刷新缓冲区。

用户统计包括今日调用量、当前 RPS/RPM、成功率、平均与 P95 耗时、缓存命中率、结果数量、状态码分布、错误类型、24 小时/7 天/30 天趋势和最近调用。

管理员统计增加全局调用量、活跃用户、调用量最高用户、限流次数、慢请求和跨用户筛选。统计查询始终带明确时间范围，避免无界扫描。

## 9. 前端设计

### 9.1 用户门户

- 未登录状态使用独立全屏登录页，不显示搜索内容。
- 登录后顶部导航包含“资源搜索、API 调用、账号设置”。
- 搜索仍是默认首页，并沿用现有 PanSou 搜索结果与筛选体验。
- API 页面展示指标卡、ECharts 趋势图、错误与状态分布、最近调用表格、Key 状态和调用示例。
- 账号设置提供修改密码、查看账号到期时间、退出登录和重置 API Key。
- 首次登录使用阻断式改密页面，不能绕过进入其他路由。

### 9.2 管理员后台

- 在现有侧边栏增加“用户管理”和“API 监控”。
- 用户管理支持创建、编辑角色与限额、设置有效期、启停、重置密码、重置或吊销 Key。
- 创建成功对话框一次性展示临时密码和 API Key，并提供复制按钮与未保存提醒。
- API 监控提供全局指标、趋势图、用户排行和调用明细筛选。

### 9.3 响应式布局

- 用户门户桌面端使用顶部导航；窄屏切换为底部导航或紧凑菜单。
- 搜索输入、筛选和结果卡片在手机端单列显示，触控目标不小于 44 像素。
- 图表根据容器宽度自适应，移动端减少坐标轴标签并支持横向查看明细表。
- 管理后台桌面端保留侧边栏；移动端使用抽屉导航，表格切换为卡片或可控横向滚动。
- 关键操作在桌面和移动端都必须可达，不以仅悬停交互承载必要信息。

## 10. 故障处理

- PostgreSQL 不可用时，无法可靠验证用户状态、API Key 和限额，因此用户与管理员受保护接口返回 `503`，不绕过鉴权继续搜索。
- 健康检查返回数据库与日志写入器状态，便于部署监控。
- API Key 无效、账号停用、账号到期、首次改密未完成、权限不足和限流分别使用稳定错误码，前端显示对应提示。
- 异步日志失败只影响监控完整性，不把成功搜索改为失败；管理员概览显示日志写入异常状态。
- 创建用户、重置密码和重置 Key 使用数据库事务，避免只完成部分凭证变更。

## 11. API 兼容性

`/api/search` 的业务参数和 `SearchResponse` 保持兼容。新增行为只包括身份要求、权限校验、限流响应头和明确错误响应。

V1 不向普通 API Key 开放 `/api/check/links`。登录、健康、管理员和内部插件接口继续使用各自权限，不纳入对外 API 产品范围。

## 12. 测试策略

### 后端单元测试

- 用户名归一化、密码校验和 `auth_version`
- API Key 生成、前缀、哈希与重置失效
- 到期、停用、软删除和角色判定
- 双层 RPS/RPM 限流及配置热更新
- `refresh=true` 权限判定
- 日志事件构建和 30 天清理边界

### PostgreSQL 集成测试

- Migration、唯一约束和初始管理员引导
- 并发创建用户与 API Key 唯一约束
- 密码重置和 Key 重置事务
- 用户隔离的日志与统计查询
- 日志批量写入和分批清理

### API 测试

- 未登录 `401`、普通用户访问管理接口 `403`
- 首次登录改密阻断
- JWT 与 API Key 搜索兼容
- 网页/API 共享限额与 `429` 响应头
- 普通用户强刷 `403`
- 停用、到期、密码重置和 Key 重置立即失效
- 数据库故障时受保护接口 `503`

### 前端端到端测试

- 用户登录、首次改密、搜索、查看日志、重置 Key和退出
- 管理员创建、启停、设置到期和调整限额
- 管理员查看全局监控，普通用户只能看到自己的数据
- 桌面、平板和手机视口下的导航、图表、表格和关键操作

## 13. 部署与迁移

1. 先执行数据库 migration。
2. 配置初始管理员凭证并启动服务。
3. 验证管理员登录、普通用户创建和首次改密。
4. 部署用户门户，将现有 `/pansou/` 入口切换为登录页。
5. 验证 Caddy 对 `/pansou/`、`/admin/` 和 `/api/*` 的代理规则。
6. 验证未登录 `401`、普通用户管理接口 `403`、超限 `429`、数据库故障 `503`。
7. 确认 30 天日志清理任务和 PostgreSQL 每日备份正常。

## 14. V1 非目标

- 用户自行注册、邮件或短信验证、忘记密码自助找回
- 多个 API Key、Key 级独立限额或权限范围
- 套餐、支付、调用余额和按量计费
- Redis、独立 API 网关、外部身份提供商或多实例共享限流
- 向普通 API Key 开放搜索以外的业务接口
- 永久保存调用明细
