# PanSou 搜索来源与多租户插件凭证设计

## 1. 背景与目标

PanSou 当前通过启动环境变量 `CHANNELS`、`ENABLED_PLUGINS` 和
`ASYNC_PLUGIN_ENABLED` 决定搜索来源。插件管理器和采集 Runner 都在进程启动时读取一次
配置，管理员无法在后台修改，也无法在不中断服务的情况下让新配置生效。

`qqpd`、`gying`、`panlian` 和 `weibo` 四个账号型插件还会把账号写入
`CACHE_PATH/*_users`，并从进程级共享账号池中选择账号。该模型无法保证用户隔离，也不能区分
普通用户私有账号、管理员内部账号和面向所有用户的共享账号。

本次改造的目标是：

- 管理员可以在后台配置 TG 渠道并启停系统内置插件，保存后立即热生效。
- 每个用户可以为管理员已启用的账号型插件配置多个独立账号。
- 普通用户优先使用自己的账号，全部失败后才回退到公开共享账号。
- 管理员搜索和自动采集优先使用管理员私有账号，再回退到公开共享账号。
- 插件密码、Cookie 和 Token 加密保存在 PostgreSQL，任何页面和 API 都不能读回明文。
- 保持单个 Go 服务，不引入 Redis、消息队列、独立插件服务或密钥服务。
- 保持现有 `/api/search` 参数、响应结构、资源库、采集任务、用户限流和 API 调用监控兼容。

## 2. 已确认的产品规则

- 搜索来源是管理员维护的全局配置，普通用户不能改变全局启用状态。
- 用户传入的现有频道或插件筛选参数只能缩小管理员已启用的来源集合，不能调用已停用来源。
- 管理端只能启停代码中已经注册的内置插件，不支持上传或执行自定义插件代码。
- 来源配置保存后立即生效；执行中的搜索不被打断，新搜索和下一个采集关键词使用新配置。
- 候选插件初始化失败时，整次来源配置更新回滚，继续使用旧配置。
- 四个账号型插件 `qqpd`、`gying`、`panlian` 和 `weibo` 在本期一次完成多租户改造。
- 每个用户可以为每个账号型插件配置多个账号，用户之间完全隔离。
- 管理员导入或登录的账号可以设为 `public_shared` 或 `admin_private`。
- 普通用户账号固定为 `user_private`，管理员不能把它改成共享账号，也不能查看或导出其凭证。
- 管理员可以查看用户私有账号的所属用户、插件、状态、到期、最近成功和最近失败，并可停用或删除。
- 普通用户只能看到共享账号的可用数量和聚合健康状态，看不到账号名称、所属人或凭证。
- 用户本人可以新增、重新登录、启停和删除自己的账号。
- 现有四个 `CACHE_PATH/*_users` 账号目录只迁移一次，迁移后的账号归类为 `admin_private`。
- 网页搜索和 API Key 搜索继续共享用户的 `3 RPS / 60 RPM` 搜索限额。
- API Key 仍只允许调用 `/api/search`，不能管理来源或凭证。
- 用户端和管理员端都必须支持桌面和移动端完整操作。

## 3. 非目标

本期不实现以下能力：

- 用户级的插件启停或自定义来源组合。
- 普通用户之间共享、转让或导出账号。
- 自动把用户私有账号提升为公开共享账号。
- 上传第三方插件二进制、脚本或插件市场。
- 独立插件服务、独立 Worker、Redis、消息队列或分布式配置广播。
- Vault、云 KMS、在线密钥轮换界面或多主密钥自动重加密。
- 对所有旧磁盘账号执行联网有效性验证；迁移只做结构和完整性校验。
- 真实第三方账号的全自动端到端测试；这类验证保留为上线烟雾测试。

## 4. 总体架构

### 4.1 新增模块

- `SourceCatalog`：代码内的内置来源目录，描述来源键、显示名称、类型、配置模式、是否需要账号及插件工厂。
- `SourceConfigStore`：读取和以版本比较方式更新 PostgreSQL 中的全局来源配置。
- `SourceSnapshotManager`：构建、验证、原子发布和回收不可变运行时来源快照。
- `CredentialStore`：持久化账号归属、状态、非敏感元数据和加密凭证。
- `CredentialCipher`：负责 AES-256-GCM 加解密、AAD 绑定和凭证指纹计算。
- `CredentialResolver`：根据请求身份、插件和健康状态生成有序的请求级凭证候选层。
- `CredentialAdapter`：封装四个插件不同的登录、验证、刷新、错误分类和搜索账号转换逻辑。
- `CredentialHealthRecorder`：记录最近使用、成功、失败、冷却和失效状态，不参与结果持久化。
- `LoginFlowStore`：保存短期二维码登录状态；仅在单进程内存中存在并带 TTL。

每个模块只通过明确接口协作。账号型插件不再自行扫描数据库或磁盘，也不再持有全局用户池；它只能使用
当前请求传入且已经授权的凭证候选。

### 4.2 搜索数据流

1. JWT 或 API Key 中间件解析用户身份，并完成用户状态、权限和共享限额检查。
2. `HybridSearchService` 按现有规则决定数据库命中、后台刷新或同步实时搜索。
3. 实时搜索开始时从 `SourceSnapshotManager` 获取一次不可变快照。
4. 用户提交的现有来源筛选与快照中的启用来源取交集。
5. 非账号型来源按现有方式执行；账号型插件向 `CredentialResolver` 请求当前身份允许的候选层。
6. 插件仅解密即将实际调用的凭证，并按插件策略执行搜索和故障回退。
7. 结果按现有规则过滤、聚合、写入共享资源库并返回兼容的 `SearchResponse`。
8. 搜索结束后释放凭证引用和来源快照，异步记录脱敏健康信息与 API 调用日志。

数据库命中后的后台刷新只捕获用户 ID，不捕获明文凭证。后台任务真正执行时重新检查用户状态并解析
该用户当前有效的凭证。自动采集和管理员发起的内部任务不冒充普通用户，使用管理员凭证选择规则。

## 5. 来源配置与热加载

### 5.1 来源配置文档

PostgreSQL 表 `search_source_configs` 保存一条固定 ID 的全局配置文档，主要字段为：

- 固定主键 `id`
- 单调递增的 `version`
- 配置结构版本 `schema_version`
- `config JSONB`
- 最近成功更新的管理员 `updated_by`
- `created_at`、`updated_at`

`config` 包含：

- 当前 `ASYNC_PLUGIN_ENABLED` 对应的总开关。
- TG 渠道数组，包括标准化渠道标识、显示名称、启用状态和排序。
- 内置插件映射，包括插件键、启用状态、排序和插件允许公开配置的字段。

插件键必须存在于 `SourceCatalog`，TG 渠道标识必须经过标准化且不能重复。插件密码、Cookie、Token
等账号凭证不能进入该 JSON。部署级密钥或不适合后台展示的系统秘密继续由部署环境提供。

`search_source_config_events` 记录脱敏的成功或失败更新事件，包括操作者、基础版本、结果、稳定错误码、
变更摘要和时间。失败事件不保存候选配置内容，也不会改变 `search_source_configs`。

### 5.2 首次初始化和兼容模式

- 配置了 `DATABASE_URL` 且表为空时，系统使用启动时解析出的 `CHANNELS`、`ENABLED_PLUGINS` 和
  `ASYNC_PLUGIN_ENABLED` 生成版本 1，保证升级前后的有效来源一致。
- 全局配置一旦生成，数据库成为唯一事实来源，后续环境变量不再覆盖后台修改。
- 未配置 `DATABASE_URL` 时继续执行旧的环境变量启动路径，不启用多用户凭证和后台动态配置。
- 账号型插件可以在没有任何账号时成功初始化，但运行状态显示“等待账号”；搜索时只跳过该插件。

### 5.3 原子更新算法

每次管理员更新携带页面读取到的 `expected_version`，服务器按以下顺序处理：

1. 验证管理员权限、JSON 结构、来源目录、字段白名单和交叉约束。
2. 获取进程级更新锁，重新读取数据库版本；版本不匹配时返回 `409`。
3. 使用完整目标配置在当前快照之外构建候选快照。
4. 为候选插件创建独立生命周期上下文，初始化并验证全部启用插件。
5. 任一初始化失败时取消候选上下文、关闭已创建实例、记录脱敏失败事件并返回错误；旧配置和旧快照不变。
6. 全部成功后在事务中以 `WHERE version = expected_version` 比较并更新配置，同时记录成功事件。
7. 事务提交后使用不可失败的原子指针操作发布候选快照，再向客户端返回新版本。
8. 标记旧快照为 retired；等它的引用计数归零后取消其生命周期上下文并关闭插件资源。

如果进程在数据库提交后、内存发布前异常退出，重启会从已提交的新配置重建快照，因此数据库仍是最终事实
来源。单进程更新锁与数据库版本比较共同防止并发管理员互相覆盖。

### 5.4 快照使用边界

- 搜索请求在实时搜索开始时 `Acquire`，所有来源结束后 `Release`。
- 采集 Runner 在每个关键词项开始时重新 `Acquire`，同一关键词执行期间不切换。
- 插件的保活、清理和连接资源都绑定快照生命周期，不能创建无法停止的永久 goroutine。
- 服务重启时必须从数据库完整构建初始快照；已提交配置无法初始化时拒绝启动，避免管理界面与运行状态不一致。

## 6. 凭证数据模型与加密

### 6.1 `plugin_credentials`

每行代表一个插件账号，主要字段为：

- 内部数字主键 `id` 和不可枚举的公开 ID `public_id`
- `plugin_key`
- `scope`：`user_private`、`admin_private` 或 `public_shared`
- 可空的 `owner_user_id`；仅 `user_private` 必须填写，其他 scope 必须为空
- 可空的 `created_by_user_id`
- 用户可识别的 `display_name`
- 严格白名单后的非敏感 `public_metadata JSONB`
- `secret_schema_version`
- `binding_fingerprint BYTEA`，绑定创建凭证时会影响账号安全域的插件公开配置
- `ciphertext BYTEA`
- `nonce BYTEA`
- `key_version`
- 不可逆的 `credential_fingerprint BYTEA`
- `owner_enabled`，由账号所有者控制
- 可空的 `admin_suspended_at`、`admin_suspended_by`，普通用户不能自行清除
- `status`：`active`、`invalid` 或 `expired`
- 可空的 `expires_at`、`cooldown_until`
- `last_used_at`、`last_success_at`、`last_failure_at`
- 脱敏的 `last_error_code`、`consecutive_failures`
- `created_at`、`updated_at`

数据库约束保证 scope 与所有者一致。有效凭证必须同时满足 `owner_enabled = true` 且未被管理员暂停。
删除普通用户时级联删除其 `user_private` 凭证；停用或到期用户时保留记录，但解析器不再返回其凭证。
删除创建管理员不会级联删除共享或管理员私有凭证。

凭证指纹使用从主密钥派生的 HMAC-SHA-256 子密钥，对适配器提取的稳定上游账号标识计算。QQ/微博使用
上游账号 ID，账号密码插件使用标准化登录名；刷新 Cookie 或 Token 不改变指纹。普通用户按
`plugin_key + owner_user_id + fingerprint` 去重，管理员拥有的两种 scope 按 `plugin_key + fingerprint`
共同去重，防止同一个管理员账号同时以私有和公开两份记录存在。指纹不能替代密文，也不能从数据库值反推出
登录名、密码或 Cookie。

### 6.2 密文格式

- `PLUGIN_CREDENTIAL_MASTER_KEY` 是 Base64 编码的 32 字节随机值。
- V1 使用 AES-256-GCM，每次写入生成独立的 12 字节随机 nonce。
- AAD 绑定 `public_id`、`plugin_key`、`scope`、`owner_user_id` 和 `secret_schema_version`，防止数据库行之间交换密文。
- 插件专属秘密以带版本的 JSON payload 加密；密码、Cookie、Token、二维码登录结果和刷新信息都属于秘密。
- 账号显示名、脱敏账号、到期时间和允许公开的账号设置才能进入 `public_metadata`。
- API 永远不返回 `ciphertext`、`nonce`、指纹或解密后的 payload；更新凭证只能整体覆盖敏感字段。
- `key_version` 为未来离线轮换保留，V1 不提供在线密钥轮换。

解密只发生在准备调用具体上游账号时。明文不进入通用插件参数、不写缓存文件、不写资源记录、不写调用日志，
也不跨请求保存在插件全局状态中。Go 无法保证字符串内存立即清零，因此实现优先使用短生命周期字节缓冲区，
避免不必要的字符串复制，并在调用结束后释放全部引用。

### 6.3 主密钥启动规则

- 数据库已有凭证、旧账号目录存在可迁移记录，或配置启用了任一账号型插件时，缺少或非法主密钥属于启动错误。
- 数据库没有凭证、没有待迁移账号且账号型插件均停用时，服务可启动，但后台明确显示凭证子系统未配置；启用账号型插件前必须配置密钥并重启。
- 启动时对所有现有密文执行 GCM 认证标签完整性校验；检测到错误密钥或损坏密文时拒绝启动，并且不得自动生成替代密钥。
- 运维必须把主密钥与 PostgreSQL 备份一起安全备份；丢失主密钥意味着已有凭证不可恢复。
- 服务未加载有效主密钥时，管理员启用账号型插件的配置更新返回 `credential_key_unavailable`，旧快照保持不变。

## 7. 凭证选择与插件执行

### 7.1 身份对应的候选层

- 普通用户网页搜索或 API Key 搜索：`user_private(当前用户)`，然后 `public_shared`。
- 管理员在用户门户中的搜索：`admin_private`，然后 `public_shared`。
- 自动采集和管理员手动采集：`admin_private`，然后 `public_shared`。
- 没有身份的兼容模式不使用 PostgreSQL 凭证，只能执行旧环境配置中的非账号来源。

候选必须同时满足插件已启用、所有者已启用、未被管理员暂停、状态为 `active`、未过期、不在冷却期、
所有者用户有效等条件。
同层账号优先近期成功且失败较少的账号，并通过单进程轮换游标避免持续压在一个账号上。

### 7.2 回退语义

- 上游返回零条结果仍属于成功，不能因为结果为空而使用共享账号重复查询。
- 私有层没有候选，或该层所有实际调用均认证失败、超时、限流或发生技术错误时，才进入共享层。
- 私有层至少有一次有效上游调用成功时，不再使用共享层，即使同层其他账号失败。
- 共享层失败只影响当前插件；其他来源继续执行。

`gying` 和 `panlian` 默认使用有序 failover，首个成功账号结束该层调用。`qqpd` 和 `weibo` 现有逻辑可能
需要多个账号贡献频道或结果，因此适配器可以在同一候选层内 fan-out；只要该层至少一个账号成功，就不进入
共享层。这是对“私有优先、全部失败才共享”规则的插件兼容实现，不允许跨层并发混用账号。

### 7.3 健康状态

- 明确的登录失效记为 `auth_failed`，将凭证状态改为 `invalid`。
- 明确到期记为 `expired`，等待用户重新登录。
- 上游限流记为 `rate_limited` 并设置短期 `cooldown_until`，不永久禁用。
- 超时和上游异常记为 `timeout` 或 `upstream_error`，增加失败计数但不直接判定凭证失效。
- 成功调用清除临时失败计数并更新最近成功时间。
- 健康状态回写失败不改变已经获得的搜索结果；记录服务端告警，由后续调用继续修正。

错误分类只能由插件适配器根据可靠的上游信号产生。未知错误不得被猜测为登录失效。

## 8. 四个账号型插件适配

### 8.1 统一适配器能力

每个 `CredentialAdapter` 明确提供：

- 插件键、公开元数据模式和秘密 payload 模式。
- 登录方式及短期登录流程。
- 新凭证验证与已保存凭证重新验证。
- 将解密 payload 转换为一次搜索调用所需的账号对象。
- Cookie 或 Token 刷新，以及刷新后重新加密持久化。
- 上游错误到稳定健康错误码的映射。
- failover 或 fan-out 执行策略。

适配器不能自行列出其他用户凭证，不能访问不属于当前候选层的账号，也不能把凭证写回磁盘。

### 8.2 登录方式

- `qqpd`：二维码登录；成功后保存完整 Cookie、脱敏 QQ 标识和账号所需的频道设置。
- `weibo`：二维码登录；成功后保存 Cookie/Token 与脱敏账号信息。
- `gying`：账号密码登录；站点 Base URL 属于管理员全局插件配置，用户凭证保存登录秘密和刷新结果。
- `panlian`：账号密码登录；保存登录秘密和 Cookie，登录失效时允许适配器在当前请求内重新登录并安全回写。

每个适配器声明哪些全局公开配置会改变凭证安全域，并据此生成 `binding_fingerprint`。例如 `gying` 的
Base URL 变化后，旧站点 Cookie 绝不能发送给新域名：适配器必须使用已保存用户名和密码对新站点重新认证，
或者把账号标记为需要重新登录。只有排序、显示名称或无关筛选变化时，凭证绑定保持有效。

二维码登录创建密码学随机 flow ID，绑定用户 ID、插件、目标 scope 和创建时间，默认数分钟后过期。轮询接口
每次重新校验当前 JWT 与 flow 所有者。服务重启会取消未完成 flow，但不影响已持久化账号。

用户名和密码只在 TLS 请求体和短生命周期验证对象中出现，不能进入访问日志或错误正文。验证成功后才创建或
覆盖数据库凭证；验证失败不产生可用账号记录。

### 8.3 旧插件路由

现有 `/qqpd/*`、`/gying/*`、`/panlian/*` 和 `/weibo/*` 管理路由不能继续提供匿名账号管理。
新实现将登录能力接入统一 JWT 鉴权 API；必要的旧页面入口只做受保护跳转，不再直接读取或写入磁盘账号池。

## 9. 旧磁盘账号迁移

迁移范围为：

- `CACHE_PATH/qqpd_users`
- `CACHE_PATH/gying_users`
- `CACHE_PATH/panlian_users`
- `CACHE_PATH/weibo_users`

迁移流程：

1. 数据库 migration 创建新表后，检查数据迁移标记和四个目录。
2. 使用各插件迁移适配器解析旧 JSON，保留可识别的状态、到期和非敏感设置。
3. 使用新的主密钥加密秘密 payload，并以 HMAC 指纹进行幂等去重。
4. 所有导入账号的 scope 固定为 `admin_private`，不自动公开共享。
5. 在一个数据库事务中写入全部凭证和 `data_migrations` 完成标记；该表以迁移键为主键，保存完成时间和脱敏数量摘要。
6. 技术错误、加密错误或格式损坏文件会回滚事务并中止启动，错误只报告脱敏文件名和原因。
7. 事务成功后数据库成为运行时唯一账号来源，插件不再加载旧目录。

空文件、与账号无关的已知辅助文件可以按迁移规则忽略；无法识别为有效格式的账号文件不能静默跳过。迁移不做
第三方联网登录，以免目标站临时故障阻塞部署；原有失效状态会被保留，后续实际使用时再更新健康状态。

旧目录不会被自动删除、覆盖或移动。系统在后台显示清理提示，管理员确认新版稳定并完成备份后自行删除。该策略
既避免不可逆数据丢失，也允许在短期内回滚到仍依赖旧目录的版本。

## 10. API 与权限

### 10.1 管理员来源接口

- `GET /api/admin/search-sources/catalog`：返回内置来源目录及公开配置模式。
- `GET /api/admin/search-sources/config`：返回当前配置、版本和运行时健康摘要。
- `POST /api/admin/search-sources/validate`：执行结构和语义校验，不发布配置。
- `PUT /api/admin/search-sources/config`：携带完整配置和 `expected_version`，执行候选初始化与原子发布。
- `GET /api/admin/search-sources/events`：分页返回脱敏更新事件。

所有接口复用现有管理员 JWT 中间件。普通用户返回 `403`，未登录返回 `401`，版本冲突返回 `409`，数据库
不可用返回 `503`。验证接口不能成为绕过保存流程的插件执行入口。

### 10.2 管理员凭证接口

管理员可以：

- 查看、新增、重新登录、启停和删除 `admin_private`、`public_shared` 凭证。
- 在这两个管理员拥有的 scope 之间改变可见性。
- 查看用户私有凭证的所属用户、插件、状态、到期、最近成功和脱敏失败。
- 停用或删除用户私有凭证。

管理员不能读取密文、指纹或秘密 payload，不能替用户重新登录，也不能把 `user_private` 改成其他 scope。
管理员暂停用户私有凭证后，用户只能查看暂停状态，不能通过自己的启用接口解除；只有管理员可以恢复。
管理员拥有的凭证在 `admin_private` 与 `public_shared` 之间切换时，因为 scope 属于 AAD，服务必须在事务内
解密并使用新 scope 重新加密，不能只修改数据库枚举值。

### 10.3 用户凭证接口

- `GET /api/user/plugins`：返回已启用账号型插件、当前用户账号摘要和共享账号聚合状态。
- `GET /api/user/plugin-credentials`：仅返回当前用户自己的非敏感账号信息。
- 凭证创建、重新登录、启停和删除接口：服务端强制 scope 和 owner，忽略客户端伪造值。
- 登录 flow 创建和轮询接口：只允许网页 JWT，flow 必须属于当前用户。

共享摘要只包含有效数量和 `available`、`degraded`、`unavailable` 等整体状态，不包含公开账号的名称、公开 ID、
最近错误或时间明细。

### 10.4 API Key 和限流

API Key 继续只允许访问 `/api/search`。来源配置、凭证、登录 flow、管理员和用户管理接口均拒绝 API Key。
网页 JWT 搜索和 API Key 搜索先归一为同一用户 ID，再共享该用户的 `3 RPS / 60 RPM` 搜索限额。

凭证管理和登录接口不消耗搜索配额。创建或完成登录默认限制为每个用户、每个插件每分钟 5 次，二维码状态轮询
限制为每个 flow 每秒 2 次；超限返回 `429` 和 `Retry-After`。所有修改接口执行后端所有权检查，不能依赖
前端隐藏按钮提供安全性。管理接口不解析 `X-API-Key`；没有网页 JWT 时统一返回 `401`，即使请求附带有效 API Key。

## 11. 管理端与用户端

### 11.1 管理端“搜索来源”

在现有 `/admin/` 增加“搜索来源”视图：

- 顶部展示配置版本、最后更新时间、当前快照状态和最近更新结果。
- TG 渠道支持新增、编辑、启停、删除、排序和重复校验。
- 内置插件展示启用状态、运行健康、是否需要账号、管理员私有数量和公开共享数量。
- 插件详情仅展示 `SourceCatalog` 允许的公开配置，例如 `gying` Base URL；敏感账号字段不在该表单出现。
- 保存时提交完整配置和当前版本；初始化失败时保留页面编辑值，明确定位失败插件，并提示旧配置仍在运行。
- 版本冲突时要求重新加载最新配置，不能静默覆盖。

### 11.2 管理端账号管理

账号管理包含两个视图：

- “管理员账号”：管理 `admin_private` 和 `public_shared`，创建时选择可见性，之后可在两者间切换。
- “用户账号监管”：按用户、插件、状态和到期筛选，只提供停用、启用和删除，不提供凭证查看或导出。

二维码和密码登录沿用统一交互状态：等待、成功、过期、失败。删除操作要求明确确认，界面不会以“查看详情”形式
泄露已保存秘密。

### 11.3 用户端“搜索账号”

用户门户增加“搜索账号”视图：

- 只列出管理员当前启用的账号型插件。
- 每个插件显示个人账号、状态、到期、最近成功、脱敏建议和共享账号聚合状态。
- 支持添加多个账号、二维码登录、账号密码登录、重新认证、启停和删除。
- 管理员停用插件后，用户账号保留但不参与搜索；重新启用后按原状态恢复候选资格。
- 用户不能查看或编辑其他用户账号，也不能看到共享账号身份。

### 11.4 响应式布局

- 管理端桌面使用现有侧边导航、表格和详情抽屉；移动端使用抽屉导航、账号卡片和底部操作面板。
- 用户端桌面可使用列表与详情面板；移动端改为单列卡片，二维码和登录状态适配窄屏。
- 所有关键操作都提供至少 44 像素触控目标，不依赖悬停，不让宽表格成为唯一操作入口。
- 页面继续使用现有 PanSou 视觉体系、ECharts 和 Lucide，不单独引入新的 UI 设计系统。

## 12. 故障处理与可观测性

### 12.1 稳定错误分类

对外和内部使用稳定错误码区分：

- `auth_failed`
- `credential_expired`
- `credential_unavailable`
- `rate_limited`
- `upstream_timeout`
- `upstream_error`
- `source_config_invalid`
- `source_init_failed`
- `source_config_conflict`
- `credential_key_unavailable`
- `database_unavailable`

普通用户只收到可操作的脱敏提示；管理员可以在来源事件和账号状态中看到插件及错误码，但仍看不到上游响应中的
敏感正文。未知上游响应不能原样透传到浏览器。

结构或参数错误返回 `400`，未登录返回 `401`，权限不足返回 `403`，不可枚举的非所属资源返回 `404`，配置
版本冲突返回 `409`，候选插件初始化失败返回 `422`，限流返回 `429`，数据库或密钥子系统不可用返回 `503`。

### 12.2 PostgreSQL 故障

数据库启用的多用户模式依赖数据库校验用户状态和 API Key。数据库不可用时：

- 已经通过鉴权并取得快照的在途请求可以完成非账号来源；无法解析的新凭证会跳过对应账号型插件。
- 新的用户和管理员受保护请求返回 `503`，不能仅凭未复核的 JWT 绕过账号停用或到期检查。
- 来源配置、凭证管理和登录 flow 持久化操作返回 `503`。
- 自动采集暂停，数据库恢复后按现有调度规则继续。
- 当前内存快照不被清空；数据库恢复后无需重启即可继续供新请求使用。

未配置数据库的兼容模式仍可使用环境变量快照执行原有非账号实时搜索，但不提供多用户凭证能力。

### 12.3 日志与健康

- API 调用日志继续按用户隔离，记录现有身份入口、耗时、状态和结果摘要。
- 插件运行日志只记录请求 ID、插件、候选 scope、耗时、结果数量和稳定错误码，不记录账号名、公开凭证 ID、密码、Cookie 或 Token。
- 来源配置事件只保存脱敏变更摘要，不保存完整凭证或请求体。
- 健康信息包含数据库、加密子系统、当前配置版本、快照状态和插件初始化状态。
- 普通健康响应不暴露内部账号数量或错误正文；详细健康仅管理员可见。
- 日志写入或凭证健康回写失败不能把成功搜索改成失败，但必须产生服务端告警计数。

## 13. 测试策略

### 13.1 单元测试

- AES-GCM 加解密、随机 nonce、AAD 行绑定、防篡改和错误主密钥。
- HMAC 指纹稳定性和不包含明文。
- scope/owner 约束、用户状态、过期、停用和冷却过滤。
- 用户启停与管理员暂停的优先级，以及用户不能解除管理员暂停。
- 普通用户、管理员和采集任务的候选层及共享回退。
- 零结果视为成功、临时错误不误判失效、明确认证错误更新状态。
- failover、fan-out、同层轮换和跨层不并发。
- 来源文档标准化、重复渠道、未知插件、字段白名单和版本冲突。
- 快照 Acquire/Release、旧快照延迟回收和候选失败清理。

### 13.2 PostgreSQL 集成测试

- 新 migration、外键、检查约束、索引和配置单例约束。
- 密文存储检查，数据库行中不存在测试密码、Cookie 或 Token 明文。
- 并发创建凭证、HMAC 去重和用户删除级联。
- 来源配置 compare-and-swap 和并发管理员冲突。
- 成功热更新、初始化失败不改变配置、事件记录脱敏。
- 四种旧目录 fixture 的完整迁移、幂等、事务回滚和完成标记。
- 数据库恢复后来源配置读取、凭证解析和采集恢复。

### 13.3 插件和服务测试

- 为四个适配器提供模拟上游，覆盖二维码等待/成功/过期、密码错误、Cookie 刷新、限流、超时和零结果。
- 修改凭证绑定的站点配置后不会把旧域名 Cookie 发送到新域名，并按适配器规则重新认证或要求重新登录。
- 两个用户并发搜索时只能收到各自候选，任何插件不能观察到另一用户凭证。
- 管理员私有账号不进入普通用户候选，用户私有账号不进入管理员或采集候选。
- 热切换期间旧搜索正常结束，新搜索使用新版本，插件 goroutine 和连接最终释放。
- 现有 `/api/search` 参数、筛选、响应、资源入库和后台刷新保持兼容。
- 现有采集任务、每渠道重试、运行摘要和冷却行为保持兼容。

### 13.4 API 与前端测试

- 未登录 `401`、普通用户管理接口 `403`、API Key 管理凭证 `401/403`、数据库故障 `503`。
- 用户不能枚举、更新或删除其他用户凭证。
- 管理员接口和数据库响应均不返回秘密字段。
- 共享摘要只包含数量和聚合状态。
- 网页与 API Key 搜索继续共享 `3 RPS / 60 RPM`。
- Playwright 覆盖管理员配置来源、热更新结果、共享/私有账号管理、用户账号登录和重新认证。
- Playwright 覆盖管理员监管用户账号但无法查看秘密，以及桌面和移动端关键流程。

验证命令包括 Go 全量测试、`go vet`、关键并发包的 race 测试、前端生产构建和 Playwright。自动测试全部使用
模拟上游，真实 `qqpd`、`gying`、`panlian` 和 `weibo` 登录作为部署后的手工烟雾测试。

## 14. 部署、迁移与回滚

1. 备份 PostgreSQL 和四个旧账号目录。
2. 生成 32 字节随机主密钥，将 Base64 值写入权限为 `600` 的部署密钥文件，再注入 `PLUGIN_CREDENTIAL_MASTER_KEY`。
3. 部署包含新增 SQL migration 的服务。
4. 启动时依次执行 schema migration、来源环境配置初始化、旧账号事务迁移和运行时快照构建。
5. 任一步失败都拒绝启动，并保留旧账号目录和旧数据库数据。
6. 启动后核对配置版本、迁移账号数量、插件健康、管理员私有数量和共享聚合可见性。
7. 手工验证四种真实登录、普通用户私有优先、共享回退、管理员采集和热更新回滚。
8. 保留旧目录直到新版稳定和备份验证完成，再由管理员手工清理。

生产环境的用户登录、二维码 flow、密码提交和所有凭证管理请求必须经过 HTTPS，反向代理不得记录请求体。

数据库变更保持向前可加，不修改旧磁盘账号。短期回滚旧二进制时，它仍可读取原目录和环境变量；新版新增表可以
暂时保留。回滚期间在新版数据库中新增或更新的凭证不会同步回旧目录，因此回滚只作为短期故障恢复手段。

## 15. 验收标准

- 管理员能够在后台维护 TG 渠道并启停所有内置插件，无需修改环境变量或重启服务。
- 成功保存后新搜索立即使用新配置；执行中的搜索不中断；失败保存不改变数据库配置或当前快照。
- 同时搜索的两个普通用户不会看到或使用对方私有账号。
- 普通用户自己的候选层全部失败后才使用共享账号；一次有效上游调用或零结果成功都不会触发共享回退。
- 管理员搜索和自动采集只使用管理员私有与公开共享账号，不使用普通用户账号。
- 管理员可以监管用户账号状态但无法查看、导出或改变其秘密和 scope。
- 管理员暂停的用户账号不能被用户自行恢复；管理员解除暂停后仍尊重用户自己的启停状态。
- PostgreSQL、API 响应、资源记录、调用日志和服务日志中不存在明文密码、Cookie 或 Token。
- 旧磁盘账号只迁移一次并归入 `admin_private`，迁移失败不会产生半完成数据。
- `/api/search` 兼容现有参数和响应，网页与 API Key 继续共享用户限额，API 调用监控仍按用户隔离。
- 管理端和用户端在桌面与移动视口下都能完成来源配置、登录、账号管理和状态查看。
