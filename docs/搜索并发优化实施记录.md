# 搜索并发优化实施记录

## 2026-07-18：P0–P2 单机架构落地

本次实现以 `docs/搜索并发优化方案.md` 为基线，保持单进程、无 Redis 的部署边界。

已落地：

- 全局出站连接池扩大到 300，总单 Host 上限 200，SOCKS5 优先使用可取消的 `ContextDialer`；
- TG 单请求 worker 改为 `TG_SEARCH_WORKERS` 配置；默认 GC 百分比调整为 75；
- 搜索热路径正则预编译，链接排序去重改为 O(n)；
- 新增进程级搜索调度器：活跃搜索 admission、有限等待队列、429 背压、TG/插件/凭据插件分池、单请求预算；
- 后台刷新与采集共用调度器，后台任务保留前台容量；
- 每来源 bulkhead、超时策略、连续失败熔断、半开探测和运行指标；
- TG 改为 `(keyword, channel)` 独立缓存，只对缓存 miss 的频道执行实时请求；
- TG 来源支持实时层/采集层，旧配置默认前 30 个频道进入实时层；
- 插件支持 primary/secondary/deep 分级，普通搜索按 15 + 30 分批执行，结果充足时提前停止；
- legacy 插件与凭据插件并行执行；
- PostgreSQL migration 019 添加 `pg_trgm` 及资源标题、正文、URL GIN 索引；数据库搜索结果封顶 1000；
- `/api/health` 与 `/api/admin/search-scheduler` 暴露调度状态；管理台来源页面可配置频道层级、来源并发和插件层级。
- live 聚合缓存与 TG 频道缓存完成预检后才按需获取 admission；纯缓存命中不占活跃搜索槽；
- 搜索响应新增 `execution`，明确 requested / executed / cached / deferred 及 tiered 策略语义；
- 来源 limiter 改为稳定的可调整 semaphore，热更新不再产生两代并发门；
- migration 020 新增每日来源指标表，运行指标定时增量落库；管理端提供 14 天来源分层建议；
- 部署脚本显式传递推荐的 `SEARCH_*` 参数，并检查 mihomo 文件句柄与 PostgreSQL shared_buffers。

## 2026-07-18：增量收尾完成

- 来源指标快照直接暴露 `total_duration_ms` 原始累计值，分钟 flush 不再使用 `AverageMS × Runs` 反推；
- TG 频道缓存增加 6 小时 stale 窗口：新鲜期到期后立即返回旧结果，并通过后台 admission 去重刷新；
- TG/插件聚合搜索对“完整且零结果”使用 10 分钟负缓存，部分结果不按负缓存处理；
- 分层建议增加资格判定：`≥100 次且 ≥14 天`，或 `≥300 次且 ≥7 天`；
- 新增管理端一键应用达标建议，使用 sourceconfig 版本乐观锁写回并热更新，未达标来源保持原层级；
- 全仓 race 检查发现并修复 `gying` 并行详情抓取共享 cloudscraper `StealthMode` 的既有竞态，按 scraper 独立加锁保护第三方可变状态。
- 增加 `SEARCH_TIERED_ROLLOUT_ENABLED` 兼容开关，默认 `false` 时保持旧版全来源实时搜索；开启后才启用 30 个 TG 实时层和插件分批策略；
- 用户端新增深度搜索入口：仅当响应 `execution.deferred > 0` 时显示，追加 `ext.deep=true` 执行完整来源搜索；
- 部署脚本改为候选镜像 + 优雅停止 + rollback 容器 + 内外部健康检查 + 自动恢复，避免 `docker rm -f` 直接打断正在运行的任务；
- 线上旧 API 保持运行期间完成 `pg_trgm` 与三个并发 GIN 索引预建，均验证为 valid/ready，未中断 API 或采集批次。

## 2026-07-18：链接检测并发优化正式上线

- `CheckService` 新增可取消的单条检测入口，队列 context 贯穿各平台 HTTP 请求，不再通过脱离 context 的 goroutine 包装检测；
- 队列总超时默认由 30 秒调整为 15 秒，各平台继续保留 8–12 秒的更短子超时；
- 迅雷 captcha token 按代理出口作用域缓存 5 分钟，使用 singleflight 合并并发刷新；token 被上游拒绝时失效并只重试一次；
- 链接检测队列改为按平台 round-robin 领取任务，默认全局 8 worker、单平台 2 并发；worker 不会因等待平台 semaphore 而占用槽位；
- 单平台连续 5 次上游错误后熔断 5 分钟，冷却后只允许一个半开探测；有效、失效、过期、提取码等业务结果不计入熔断；
- due 候选每分钟继续拉取，精确 backlog count 与 ETA 采样独立降为每 5 分钟；候选最多扩取至 1000 条后按平台交错执行；
- 线上约 27 万资源、`unknown` 每小时复检策略下，due count 的 `EXPLAIN (ANALYZE, BUFFERS)` 实测约 587ms、shared buffers read 约 9542，验证了独立降频的必要性；
- 内存检测缓存、迅雷 token 缓存与 bbolt 缓存增加每小时定时清理，并随 `CheckService.Close` 正常停止；
- 检测结果默认每 16 条或 1 秒批量写入一个 PostgreSQL 事务，逐行保留 `SELECT FOR UPDATE` 和负面状态二次确认语义，批次任一失败时整体回滚；
- 新增平台并发、熔断半开、批量写回、backlog 降频、token singleflight/失效重试、context 取消及缓存清理测试。
- 通过 `scripts/deploy/update-backend.ps1` 完成后端滚动发布，候选镜像为 `local/pansou-api:20260718105214`；前端本轮无改动，未重复发布；
