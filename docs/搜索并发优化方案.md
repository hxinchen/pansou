## 2. 上线闭环清单（必做）

### 2.1 全量测试 + race 检查

```bash
GOTOOLCHAIN=go1.24.9 go build ./...
GOTOOLCHAIN=go1.24.9 go test -race ./...
```

调度器是本次唯一新增的并发原语，race 干净是上线前置条件。

### 2.2 部署与冒烟

低峰执行 `scripts/deploy/update-backend.ps1`。脚本会先在旧容器在线期间执行 `pg_trgm` 与三个 `CREATE INDEX CONCURRENTLY`，验证索引 valid/ready 后才切换；切换使用 SIGTERM + 30 秒宽限，旧容器保留为 rollback 容器，内网和公网健康检查均通过后才清理。migration 019/020 仍会自动执行，已预建索引会快速跳过。

首次上线默认 `SEARCH_TIERED_ROLLOUT_ENABLED=false`，保持旧版全来源实时搜索；确认 API 与前端稳定后，再通过部署参数或容器环境变量显式开启分层。

上线后冒烟三件事：

1. **功能**：搜一个冷关键词，确认响应含 `execution` 字段、`/api/admin/search-scheduler` 计数合理；
2. **背压**：并发压 15 个不同冷词，确认超出容量的请求收到 429 + `Retry-After`，`peak_active` 封顶在 8；
3. **索引生效验证**（最容易翻车的一处）：

```sql
EXPLAIN ANALYZE SELECT id FROM resources WHERE title ILIKE '%凡人修仙传%';
EXPLAIN ANALYZE SELECT id FROM resources WHERE title ILIKE '%狂飙%';
```

`ILIKE '%kw%'` 走 GIN 索引依赖 trigram 选择率，**2 个汉字的短关键词可能被 planner 判定不如顺序扫描**。若短词不走索引：记录进实施记录，后续考虑表达式索引或限制 content 检索范围，不阻塞本次上线。

### 2.3 实测数据回填

把冒烟与压测数字写进 `docs/搜索并发优化实施记录.md`，作为后续自动分层与 P3 决策的基线。

## 3. 增量优化项（已完成）

### 3.1 TG 频道缓存 stale-while-revalidate ✅

现状：per-channel 缓存 60 分钟 TTL 到期即硬 miss，热关键词每小时仍要全量扇出一次、该次用户吃慢请求。

实现：频道缓存记录独立新鲜期，并额外保留 6 小时 stale 窗口；过期但存在时**先返回旧结果**，同一 `(keyword, channel)` 只提交一个后台刷新任务。后台任务使用 `PriorityBackground` admission，不占前台预留容量。

### 3.2 无结果负缓存 ✅

TG 与插件聚合搜索对“complete 且 0 结果”使用 10 分钟 TTL；部分完成的空结果不进入负缓存语义。

### 3.3 管理台“一键应用分层建议” ✅

新增应用接口与管理台按钮，按 sourceconfig 乐观锁版本写回并热生效。后端硬性过滤未达标来源，前端明确展示运行次数、观察天数和资格状态。

### 3.4 指标精度小修 ✅

`SourceSnapshot` 已直接暴露 `TotalDurationMS`，flush 使用原始累计值计算增量，不再通过整数平均值反推。

## 4. 自动分层的数据标准

### 4.1 为什么现在不能切自动分层

1. **指标从零开始**：`search_source_metrics_daily` 随本次上线才开始积累；
2. **比率指标的小样本噪声**：打分依赖成功率、超时率、独有结果率。一个频道只跑 3 次、失败 1 次，会显示 33% 失败率——这是噪声不是信号；
3. **流量周期性**：工作日/周末关键词构成不同、上游站点有维护窗口、新资源发布有波峰。短窗口会把偶然当规律。

### 4.2 达标标准（硬性）

一个来源参与自动分层决策，需**同时**满足：

| 维度 | 阈值 | 依据 |
|---|---|---|
| 单来源运行次数 | ≥ 100 | 成功率估计达到 ±10% 精度（95% 置信）约需 `0.25×(1.96/0.1)² ≈ 96` 个样本；P95 延迟同样需要百级样本 |
| 时间窗口 | ≥ 14 天 | 覆盖两个完整周循环（工作日/周末差异、上游维护窗口） |

补充规则：

- 单来源样本 ≥ 300 时，可提前到 **7 天**决策（样本量弥补窗口不足）；
- 未达标来源**维持当前层级**，不强行打分；
- 采集层长尾频道只靠 Runner 产生流量，14 天攒不够 100 次的，保持采集层即为正确结论，无需决策；
- 分层调整始终经管理台人工确认（3.3 的一键应用），不做无人值守自动切换。

### 4.3 进度查询

```sql
SELECT source, sum(runs) AS runs, min(metric_date) AS since
FROM search_source_metrics_daily
GROUP BY source
ORDER BY runs DESC;
```

| 项目 | 触发条件 |
